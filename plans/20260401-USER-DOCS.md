# User-facing documentation plan

Goal: generate user-facing documentation for the Slope app from markdown source files and PNG screenshots, with a bias toward a zero-build static approach that fits the current repo architecture.

## Context
- The app is a static, zero-build site using CDN-loaded libraries and ES modules.
- Existing user-facing markdown is centered in `slope.md`.
- Existing screenshot coverage is minimal; `docs/full-app-edit.png` is the only current documentation screenshot.
- Internal dated prompt files in `prompts/` are implementation notes, not end-user docs.

## Recommended docs shape
- Create a standalone docs surface rather than embedding docs inside the main map UI first.
- Use markdown files as the source of truth.
- Store screenshots as PNG assets in a dedicated docs asset folder.
- Render markdown client-side in a static docs page to preserve the repo's no-build deployment model.

## Why standalone docs page first
- Lowest coupling to the map app runtime, URL hash sync, and panel layout constraints.
- Easier to iterate on information architecture, screenshots, and navigation.
- Keeps the app focused on usage and the docs focused on onboarding, features, and workflows.
- Can still add a lightweight in-app `Help` link later.

## Client-side markdown rendering options

The practical question is not just which parser is best in general, but which one matches this repo's delivery model:
- static hosting,
- no bundler,
- trusted markdown files stored in the repo,
- PNG screenshots referenced with relative paths,
- and a docs shell that should stay easy to inspect and maintain.

At a high level, the browser-side architecture for all three serious options looks like this:

```html
<!-- docs/index.html -->
<nav id="docs-nav"></nav>
<main id="docs-content"></main>

<script type="module">
  const response = await fetch('./content/intro.md');
  const markdown = await response.text();
  // render markdown here
  document.getElementById('docs-content').innerHTML = renderedHtml;
</script>
```

The difference is what sits in the `render markdown here` step and how much transformation power you want before injecting HTML into the page.

### Marked
Pros:
- Small, fast, and simple.
- Easy CDN usage in a zero-build static page.
- Good fit when markdown is mostly standard prose, headings, lists, images, tables, and code blocks.
- Minimal setup and low maintenance overhead.

Cons:
- Less structured processing pipeline than the unified/remark ecosystem.
- Advanced transforms, linting, AST rewrites, or content plugins are less ergonomic.
- Sanitization must be handled explicitly if rendering untrusted markdown.

Best fit:
- Static app docs with trusted local markdown files and simple rendering needs.

Implementation shape:
- Best when `docs/index.html` fetches one markdown file at a time and renders it directly.
- Very low code volume.
- Easy to combine with a tiny route model such as `?page=track-editing` or `#track-editing`.

Example:

```html
<script src="https://cdn.jsdelivr.net/npm/marked/lib/marked.umd.js"></script>
<script type="module">
  const pages = {
    intro: './content/intro.md',
    editing: './content/track-editing.md',
    profile: './content/profile.md'
  };

  async function renderCurrentPage() {
    const hash = window.location.hash.replace(/^#/, '') || 'intro';
    const path = pages[hash] || pages.intro;
    const markdown = await fetch(path).then((response) => response.text());
    const html = marked.parse(markdown);
    document.getElementById('docs-content').innerHTML = html;
  }

  window.addEventListener('hashchange', renderCurrentPage);
  renderCurrentPage();
</script>
```

Notes:
- This is the cleanest fit for the current repo.
- If needed, add DOMPurify in front of `innerHTML`.
- If docs need a generated table of contents later, that can be added with a small custom pass or a plugin, but that is not the natural strength of `marked`.

### Remark / unified
Pros:
- Strong AST-based pipeline.
- Excellent if docs need plugins, custom directives, linting, frontmatter processing, table-of-contents generation, reusable transforms, or MDX-like future growth.
- More extensible long term.

Cons:
- Heavier mental model and setup cost.
- Less attractive for a simple zero-build browser-only page unless there is a clear need for the ecosystem.
- More moving parts than this repo currently warrants.

Best fit:
- Larger docs systems, content pipelines, pre-processing, or future docs tooling expansion.

Implementation shape:
- Best when docs are not just rendered, but processed.
- Useful if you want structured transforms such as:
  - rewriting image URLs,
  - injecting heading anchors,
  - validating or reshaping content,
  - supporting frontmatter,
  - generating a TOC from headings,
  - or introducing custom markdown directives.

Browser-oriented example:

```html
<script type="module">
  import { unified } from 'https://esm.sh/unified@11';
  import remarkParse from 'https://esm.sh/remark-parse@11';
  import remarkGfm from 'https://esm.sh/remark-gfm@4';
  import remarkRehype from 'https://esm.sh/remark-rehype@11';
  import rehypeStringify from 'https://esm.sh/rehype-stringify@10';

  async function renderPage(path) {
    const markdown = await fetch(path).then((response) => response.text());
    const file = await unified()
      .use(remarkParse)
      .use(remarkGfm)
      .use(remarkRehype)
      .use(rehypeStringify)
      .process(markdown);

    document.getElementById('docs-content').innerHTML = String(file);
  }

  renderPage('./content/intro.md');
</script>
```

Notes:
- This is more powerful, but noticeably more architectural weight.
- The power comes from the pipeline. If there is no planned pipeline, most of that power goes unused.
- In this repo, `remark` would make sense only if the docs plan is likely to expand into a more formal documentation system.

### Markdown-it
Pros:
- Also simple and fast.
- Strong plugin ecosystem.
- Often a good middle ground between `marked` simplicity and `remark` extensibility.
- Easy browser usage.

Cons:
- Still less structured than remark for deeper content processing.
- Plugin choices can become ad hoc over time.

Best fit:
- Static browser-rendered docs where a few plugins are useful but a full unified pipeline is unnecessary.

Implementation shape:
- Good middle-ground choice if you want a simple browser renderer but expect a few enhancements such as anchors, attributes, containers, or syntax tweaks.
- Easier to keep zero-build than `remark`, while giving more extension room than `marked`.

Example:

```html
<script src="https://cdn.jsdelivr.net/npm/markdown-it/dist/markdown-it.min.js"></script>
<script type="module">
  const md = window.markdownit({
    html: false,
    linkify: true,
    typographer: true
  });

  async function renderPage(path) {
    const markdown = await fetch(path).then((response) => response.text());
    document.getElementById('docs-content').innerHTML = md.render(markdown);
  }

  renderPage('./content/intro.md');
</script>
```

Notes:
- If the docs need just a little more than `marked`, this is the main alternative.
- It is still not the right tool if the docs roadmap includes AST-level content transformation or validation.

### Showdown / others
Pros:
- Lightweight legacy options exist.

Cons:
- Weaker default recommendation today versus `marked` or `markdown-it`.
- Less compelling unless there is an existing project constraint.

Best fit:
- Not preferred here.

## Architecture patterns for this repo

Three realistic architectures fit the current codebase.

### Option A: standalone docs app, client-rendered markdown

Structure:

```text
docs/
  index.html
  assets/
    overview-map.png
    track-panel.png
  content/
    intro.md
    track-editing.md
    profile.md
  js/
    docs.js
  css/
    docs.css
```

Flow:
1. `docs/index.html` loads a tiny docs script.
2. The script maps a route key to a markdown file.
3. The markdown file is fetched and rendered.
4. Relative image links in markdown point to `../assets/...` or `/docs/assets/...` depending on final layout.

Why it fits:
- Completely separate from map runtime concerns.
- No coupling to `index.html`, map state, or app panels.
- Easy to host statically.

Minimal module sample:

```javascript
// docs/js/docs.js
const pages = {
  intro: './content/intro.md',
  editing: './content/track-editing.md',
  profile: './content/profile.md'
};

export async function loadDocsPage(pageId, renderMarkdown) {
  const path = pages[pageId] || pages.intro;
  const markdown = await fetch(path).then((response) => response.text());
  return renderMarkdown(markdown);
}
```

### Option B: render `slope.md` directly as version 1

Structure:

```text
docs/
  index.html
  js/docs.js
slope.md
docs/full-app-edit.png
```

Flow:
1. Docs shell fetches `../slope.md`.
2. It renders a single large page.
3. Later, content can be split without changing the rendering model.

Why it fits:
- Fastest path.
- Avoids early content restructuring.

Tradeoff:
- `slope.md` currently mixes user-facing and engineering-oriented detail, so this works best as a transition step, not the polished end state.

### Option C: in-app docs panel

Structure:
- `index.html` gets a docs panel or modal.
- `js/docs.js` is added to the app runtime.
- Hash or query state decides which doc page is open.

Why it is weaker for phase 1:
- It competes with profile, track, and settings panels.
- It introduces URL-state interaction with an app that already uses hash parameters.
- It is more work for less documentation value than a separate docs page.

Use this only if the primary goal is contextual help from inside the app, not publishable user documentation.

## Concrete recommendation on tool + architecture pairing

Best pairing for phase 1:
- Tool: `marked`
- Architecture: standalone docs app under `docs/`

Why this pairing is the strongest:
- smallest implementation,
- minimal cognitive load,
- aligned with the repo's zero-build philosophy,
- easy to inspect in plain HTML/JS,
- and easy to evolve later into split markdown pages or even a pre-rendered pipeline.

Reference implementation sketch:

## Recommendation for this repo
Primary recommendation: use `marked` for the first version.

Why:
- The repo already favors a static, low-complexity, zero-build architecture.
- The likely input is trusted local markdown plus local PNG screenshots.
- The immediate need is rendering, navigation, and presentation, not a complex markdown processing pipeline.
- `marked` keeps implementation short, understandable, and aligned with the current app philosophy.

Fallback recommendation:
- Use `markdown-it` instead if you already know you want a couple of presentation-oriented plugins soon.
- Defer `remark` unless there is a real need for AST transforms, content linting, or a more serious docs toolchain.

## Sanitization and trust model
- If docs are loaded only from repository-controlled markdown files, client-side rendering with `marked` is reasonable.
- If any user-supplied markdown will ever be rendered, add sanitization explicitly, for example with DOMPurify.
- Even with trusted markdown, define a narrow allowed feature set and avoid raw HTML in docs unless necessary.

## Should server-side rendering make more sense?
Short answer: not for the first version.

### Client-side rendering makes more sense when
- The site is static and should remain deployable with no server logic.
- Docs content is local and limited in size.
- SEO is not the primary driver.
- Fast iteration and minimal tooling matter more than perfect first-paint HTML.
- You want to keep docs hosting as simple as `serve`, GitHub Pages, or any static host.

### Server-side or pre-rendered HTML makes more sense when
- SEO and social previews matter significantly.
- You want excellent no-JS rendering and indexing.
- Docs volume grows and navigation/search/metadata become more sophisticated.
- You want automatic TOC generation, cross-link validation, frontmatter pipelines, and content checks.
- You are willing to add a build step or a docs generator.

### Opinion for this use-case
- Start client-side.
- Prefer a standalone static docs page that fetches markdown files and renders them in-browser.
- Revisit pre-rendering only if docs become public-facing enough that SEO, indexing, or performance on very slow devices becomes important.
- If the project later needs pre-rendering, migrate to a lightweight static generation flow rather than a true server requirement.
