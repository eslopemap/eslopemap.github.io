# Tauri Spike 1 — Protocol vs Localhost MBTiles Serving

## Goal

Decide whether offline/base-map tile serving in the Tauri desktop app should use:

- `localhost` HTTP served by Rust
- a Tauri custom protocol

The spike must compare both approaches with the **same frontend behavior**, the **same dummy MBTiles dataset**, and a **repeatable validation checklist**.

## Decision to Validate

Primary candidate:

- **`localhost` HTTP first**

Secondary candidate:

- **custom protocol only if it shows a real advantage after measurement**

## Deliverables

Create two minimal demonstrator apps under `spike_demo/`:

- `spike_demo/mbtiles_to_localhost`
- `spike_demo/mbtiles_custom_protocol`

Each demonstrator must:

- be a minimal Tauri app
- use only `@eslopemap/maplibre-gl` as an online dependency
- support switching between:
  - an online OSM raster source
  - a local MBTiles raster source
- use the same tiny deterministic dummy MBTiles file
- expose enough logs and diagnostics to compare behavior

## Non-Goals

- no PMTiles in this spike
- no GPX sync in this spike
- no production UI styling
- no cache eviction logic
- no persistence beyond what is needed for the demo

## Shared Demo Requirements

### Frontend

Both demonstrators must use the same minimal frontend design:

- one page
- one map instance
- one status panel
- one source switch control:
  - `Online OSM`
  - `Offline MBTiles`
- one small debug panel showing:
  - current source mode
  - last tile URL requested
  - number of tile requests
  - last tile error, if any

### Dummy Tile Dataset

Create a tiny raster dataset for z1/z2/z3 only.

Each tile PNG should contain:

- a solid background color by zoom level
- a visible square border
- centered text with:
  - zoom `z`
  - tile `x`
  - tile `y`

Example text:

```text
z=2
x=1
y=3
```

Requirements:

- deterministic output
- committed into the repo
- generated by script, not by hand
- used for both demo and automated checks

### MBTiles Fixture

Create one fixture MBTiles file from the generated PNGs.

Suggested location:

- `tests/fixtures/tiles/dummy-z1-z3.mbtiles`

Also commit the tile source images and the generation script so the fixture can be rebuilt.

Suggested supporting files:

- `tests/fixtures/tiles/src_png/...`
- `tests/fixtures/tiles/build_dummy_mbtiles.py`
- `tests/fixtures/tiles/README.md`

When generating with Python locally, use `python3`.

## Demonstrator A — `mbtiles_to_localhost`

### Architecture

- Tauri window loads the shared minimal frontend
- Rust starts a loopback-only HTTP server on `127.0.0.1:<port>`
- MapLibre offline raster source points to:

```text
http://127.0.0.1:<port>/tiles/dummy/{z}/{x}/{y}.png
```

### Rust Responsibilities

- open the dummy MBTiles file
- convert XYZ to TMS row lookup
- read tile bytes from SQLite
- return `image/png`
- return `404` when tile missing
- expose debug logs per request

### Tauri Requirements

- use official `tauri-plugin-localhost`
- keep the port explicit or logged clearly
- expose base URL to the frontend via app-owned bootstrap config

### Validation Focus

- does MapLibre request tiles normally
- do tiles render without hacks
- are errors understandable
- does source switching remain stable
- do logs and MIME behavior stay simple

## Demonstrator B — `mbtiles_custom_protocol`

### Architecture

- Tauri window loads the shared minimal frontend
- Rust registers a custom protocol
- MapLibre offline raster source points to:

```text
mbtiles-demo://dummy/{z}/{x}/{y}.png
```

### Rust Responsibilities

- same tile lookup behavior as Demonstrator A
- same MBTiles file
- same MIME responses
- same missing-tile behavior
- same debug logging

### Validation Focus

- does MapLibre accept the custom scheme cleanly
- do raster tiles render reliably
- are requests and failures easy to reason about
- are there any unexpected differences in caching, worker behavior, or error handling

## Shared Acceptance Checklist

Both demos must pass all checks below.

### Functional

- app launches successfully in Tauri dev mode
- online OSM source renders
- offline MBTiles source renders
- switching from online to offline works repeatedly
- switching from offline to online works repeatedly
- missing tiles fail gracefully without freezing the map
- pan/zoom across available and unavailable areas remains stable

### Observability

- last requested tile path is visible
- tile request count increments predictably
- errors are surfaced in the debug panel and Rust logs
- logs are easy to compare across both demos

### Technical

- content type is correct for PNG
- z/x/y translation is correct
- TMS y-flip is correct
- offline source works without network access
- startup does not require any external tile service except OSM for the online mode

## Comparison Criteria

At the end of the spike, compare the two demos on:

- **MapLibre compatibility**
- **implementation complexity**
- **debuggability**
- **MIME/header correctness**
- **error clarity**
- **future support for vector tiles**
- **future support for caching and instrumentation**
- **testability**

## Recommended Test Strategy

### Unit-Level

- Rust tests for XYZ→TMS conversion
- Rust tests for MBTiles lookup
- Rust tests for MIME selection
- Rust tests for missing-tile behavior

### Integration-Level

- start each server/protocol implementation
- request a known tile
- assert `200` + PNG bytes
- request a missing tile
- assert `404`

### End-to-End

Use rodney with `uvx rodney --help`, Playwright, or equivalent scripted checks against the demonstrator UI:

- switch to offline mode
- wait for tile requests
- assert map canvas rendered
- collect logs on failure

## Exit Criteria

Recommend **localhost-first** unless the custom protocol demonstrator is clearly better on multiple criteria and shows no MapLibre-specific downsides.

The spike is complete only when:

- both demonstrators exist
- both use the same fixture dataset
- both can be run by another developer without guessing
- the comparison matrix is filled in
- a final recommendation is written at the bottom of the spike results
