Lit/LitElement analysis --- would it make sense?
----------------------------------------------

**Short answer: No, not for this app.**

Here's why:

| Factor | Assessment |
| --- | --- |
| **Rendering model** | This app's UI is MapLibre GL (WebGL canvas) + a few HTML controls. Lit's value is reactive DOM templating --- but 90% of the visual output is MapLibre's canvas, not DOM nodes. The settings panel, track list, and profile are the only DOM-heavy parts. |
| **Component tree depth** | Shallow. There are ~5 UI regions (toolbar, settings, track panel, profile, legend). A framework shines when you have deep nesting and data flowing through many layers. Here, one flat module handles each region. |
| **State complexity** | Medium. The [state](vscode-file://vscode-app/Applications/Visual%20Studio%20Code.app/Contents/Resources/app/out/vs/code/electron-browser/workbench/workbench.html) object + track editor variables need coordination, but a simple Proxy store (already scaffolded) handles this well. Lit's `@property` reactivity would add a framework layer without simplifying the actual state machine. |
| **Zero-build constraint** | Lit works from CDN (`esm.sh/lit`), so technically compatible. But it adds ~16 KB. More importantly, it imposes a component-class structure that doesn't match the app's imperative MapLibre integration pattern. |
| **Shadow DOM conflicts** | Lit uses Shadow DOM by default. MapLibre popups, markers, and the profile canvas would need to pierce the shadow boundary or use `createRenderRoot() { return this; }` to opt out --- adding complexity for no gain. |
| **Testing** | Current tests use simple DOM queries (`#map`, [.track-item](vscode-file://vscode-app/Applications/Visual%20Studio%20Code.app/Contents/Resources/app/out/vs/code/electron-browser/workbench/workbench.html)). Lit Shadow DOM would break these or require light DOM mode everywhere. |

**Better alternatives if the UI grows:**

-   **Stay vanilla** --- the current module split (utils.js, future dem.js, tracks.js, profile.js, ui.js) keeps concerns separated without a framework tax. For the ~100 DOM elements in this app, [document.createElement](vscode-file://vscode-app/Applications/Visual%20Studio%20Code.app/Contents/Resources/app/out/vs/code/electron-browser/workbench/workbench.html) + [innerHTML](vscode-file://vscode-app/Applications/Visual%20Studio%20Code.app/Contents/Resources/app/out/vs/code/electron-browser/workbench/workbench.html) is fine.
-   **Preact/htm** (3 KB, no build step) --- if you eventually want JSX-like templating for the track list or settings panel, Preact with `htm` tagged templates gives you a component model without Shadow DOM overhead.
-   **nanostores** (300 bytes) --- if you want reactive state without the Proxy scaffolding, nanostores integrates with vanilla or Preact and is trivially small.

**Bottom line:** Lit is a great library for building reusable web components with complex UIs. This app is a map viewer with a thin control layer --- the complexity is in WebGL shaders, DEM processing, and MapLibre interactions, not in DOM management. The vanilla module split you now have is the right level of abstraction.
