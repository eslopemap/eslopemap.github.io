REFINEMENT
======

## 1. Product framing

This feature set should be delivered as a consolidation of the current track workspace into a real GPX workspace, not as a pile of independent buttons.

The current codebase already has:

- a floating track panel with inline rename support at track and group level
- GPX import/export with tracks, routes, waypoints, timestamps, and multi-segment tracks
- a profile panel with speed and pause display
- a flat track array plus a lightweight two-level grouping model

The next step is to replace the ad hoc grouping model with a proper tree model and a shared operation layer.

## 2. Research-backed constraints

### GPX 1.1 constraints

- GPX has `metadata`, `wpt`, `rte`, and `trk` at file level.
- `trk` supports `name`, `cmt`, `desc`, `src`, `link`, `number`, `type`, `extensions`, and one or more `trkseg`.
- `rte` supports `name`, `cmt`, `desc`, `src`, `link`, `number`, `type`, `extensions`, and one or more `rtept`.
- `trkseg` has only `trkpt` and optional `extensions`. It has no native `name` field in GPX 1.1.
- `trkpt` and `rtept` inherit waypoint-like point fields, but exposing full point metadata editing in the first iteration is not justified by the current UI.
- GPX has no native `folder` or `file tree` concept across multiple files. `folder` and `file` are therefore workspace objects, not GPX-native objects.

Implication: `folder` and `file` can be renamed in-app, `trk` and `rte` can be renamed in GPX, `trkseg` must stay auto-labeled.

### Algorithm choices

- For line simplification, Visvalingam-Whyatt is the better default for natural GPS traces because it produces less angular results than Douglas-Peucker. Douglas-Peucker remains useful as an advanced compatibility option because users understand tolerance-in-meters.
- For noisy speed signals, plain moving average is simple but sensitive to spikes and pause boundary leakage. Rolling median is more robust to outliers. The best fit here is a hybrid: time-bin first, rolling median second, optional short moving average last.

## 3. Refined object model

Introduce a tree workspace model with explicit node types:

- `folder`
- `file`
- `track`
- `segment`
- `route`
- `waypoint`

Point-level objects ( `trkpt` - `rtept` ) exist in the data model but are not rendered in the tree by default, it would be unusable on real GPS files. 

### Hierarchy rules

- Imported folder: `folder > file > track > segment`
- Imported standalone GPX: `file > track > segment` or `file > route`
- Imported waypoints live under `file`
- Newly created content defaults to a synthetic unsynced file if no file exists yet

### Naming rules

- `folder`: editable
- `file`: editable, and only file creation/import triggers the naming popup
- `track`: editable
- `route`: editable
- `segment`: not editable in V1, shown as `Segment 1`, `Segment 2`, etc.
- `waypoint`: editable in V1 only when pause cleanup creates it or when selected from the map

### Default naming

- New folder: `New folder`
- New file: `Outing 1.gpx`
- New track: Basename of the file + `#1`, `#2`, ...
- New route: `Route 1`, `Route 2`, ...
- New waypoint: `Pause 1` or `Waypoint 1` depending on origin
- New segment: `Segment N` display label only

## 4. Tree UX

### Interaction model

- The tree stays compact by default.
- Each row has: disclosure, color/icon, name, stats summary, kebab menu.
- A small `+` button appears on hover or on row selection for levels that can contain children.
- Primary object actions come from a context menu on right click, plus a long-press equivalent on touch devices.
- Bulk selection is hidden by default behind a `Select` action in the track-actions menu.
- In select mode, checkboxes appear for eligible rows except `segment` if the action does not apply.

### Context menu

The object context menu becomes the main entry point for metadata editing and object actions. It should be available from:

- right click on desktop
- long press on touch
- kebab button for discoverability

Baseline menu:

- `Info`
- `Appearance...`
- `Hide`
- `New segment` when the target is a track
- `Select all`
- `Center`
- `Duplicate`
- `Copy`
- `Cut`
- `Paste`
- `Delete`

`Info` opens a side sheet or compact modal for metadata editing.

V1 `Info` fields by object type:

- `file`: name, desc
- `track`: name, desc, cmt, type
- `route`: name, desc, cmt, type
- `waypoint`: name, desc, cmt, sym, type
- `folder`: name only in V1
- `segment`: no Info editor in V1, because GPX 1.1 has no native segment naming field

### Add-child rules

- `folder` can add `file`
- `file` can add `track`, `route`, `waypoint`
- `track` can add `segment`
- `route` cannot add child objects from the tree

### Sync affordance

- If a folder was opened with File System Access API and handles are retained, show a sync badge on the folder row.
- Badge states:
  - `linked`: two-way sync possible
  - `read-only`: imported from drag/drop or webkitdirectory fallback
  - `dirty`: local changes not yet written
- Tooltip copy should explain capability, not implementation detail.

## 5. Metadata editing UX

Rename should no longer depend on double-click. Double-click is too collision-prone in a tree that also needs expand, select, center, and edit gestures.

Acceptance criteria:

- `Info` from the context menu is the canonical way to edit name and description fields.
- The same `Info` action must be reachable from keyboard via a shortcut and from touch via long press.
- Enter commits, Escape cancels, blur commits when editing inline fields inside the Info sheet.
- Empty names revert to previous value.
- Newly created files open `Info` immediately after creation so naming still happens at the right moment.
- Context menu actions must not trigger row selection toggle, expand/collapse, or map fit unless that action was explicitly chosen.

## 6. Shared operations menu

All geometry actions move into one reusable operation layer, applicable to one or multiple selected objects.

### Unified tool interface

Track tools and selection tools should converge on one normalized operation contract so the same action can run on:

- a full track
- a segment
- a selected continuous span inside a track
- a rectangle-derived selected span

Inspiration:

```text
tool(track, statSeg, startIdx, endIdx, options)
```

Refined V1 contract:

```text
tool(track, selectionSpan, startIdx, endIdx, options)
```

Where:

- `track`: the source track object
- `selectionSpan`: a derived summary object for the active working span, including stats and geometry bounds
- `startIdx`, `endIdx`: inclusive indices on the source track
- `options`: tool-specific parameters

Rules:

- Full-track actions call the same tool with `startIdx = 0` and `endIdx = track.coords.length - 1`.
- Segment actions call the same tool with the segment-local span mapped back to the underlying track storage.
- Rectangle selection resolves first to the smallest continuous enclosing span, then invokes the same tool contract.
- Tool previews and consequence labels should be based on `selectionSpan`, not recomputed ad hoc in each UI path.

### Eligible target rules

- `track` operations apply to `track`, `segment`, `file`, and `folder` by expansion to descendant tracks.
- `route` operations apply to `route`, `file`, and `folder` by expansion to descendant routes.
- Mixed selections are allowed only when the action semantics are clear.

### 6.1 Convert route to track

Menu item: `Convert to track`

Rules:

- Input: one or more routes
- Output: one track per route, with one segment per route
- Preserve: `name`, `desc`, `cmt`, `type`, point coordinates, elevation, timestamps when present
- Keep original route by default and create sibling track
- Secondary action: `Convert and replace`

Reason: route semantics and track semantics are different enough that destructive replacement should not be default.

### 6.2 Simplify track

Menu item: `Simplify...`

Default algorithm:

- Visvalingam-Whyatt
- Threshold unit: effective area derived from a user-facing geometry tolerance in meters
- Elevation guardrail implemented as a veto layer, not by rewriting the base simplification algorithm

Advanced option:

- Douglas-Peucker with tolerance in meters

Guardrails:

- Never remove first or last point of a segment
- Never remove a local elevation extremum that is materially higher or lower than its neighbourhood
- Neighbourhood should be evaluated with a lightweight window: the closest 5 points before and after, capped by a horizontal span target of about 20 to 30 m on each side
- If that hybrid neighbourhood is expensive to maintain, fall back to a symmetric 5-point window in V1
- Never leave adjacent surviving points farther apart than `15 * horizontalTolerance`
- Never merge segments
- Never simplify across pause-created segment boundaries
- Show preview counts: `12,438 points -> 3,121 points`

Algorithm choice note:

- Visvalingam-Whyatt is easier to adapt for this requirement because it deletes points iteratively and can cheaply skip points flagged as protected extrema.
- Douglas-Peucker can still be offered as an advanced option, but adapting it cleanly to a hybrid distance plus elevation-preservation model is less straightforward because its keep-or-drop decision is driven by recursive geometric error, not local vertex salience.
- The max-gap guardrail also fits Visvalingam-Whyatt better in practice because candidate deletions can be vetoed locally when they would create an overlong retained segment.

UX:

- Modal with live preview stats, not live geometry redraw in V1
- Presets: `Light`, `Medium`, `Strong`, `Custom`

### 6.3 Split / merge tracks

Menu group: `Split / merge`

Split variants:

- `Split at selected point`
- `Split at all detected pauses`
- `Split every N km`
- `Split into segments only` versus `Split into separate tracks`

Merge variants:

- `Merge selected tracks as segments of one track`
- `Merge selected segments into one segment`

Rules:

- Merge only siblings under the same file in V1
- Preserve chronological order if timestamps exist, otherwise preserve current tree order
- If endpoints are farther apart than a warning threshold, show a confirmation
- Do not silently reverse geometries in V1; if direction mismatch is detected, offer an explicit `Reverse before merge`

## 6.4 Shortcuts

Keyboard shortcuts should cover visibility and the most frequent actions, while avoiding browser conflicts when possible.

Core shortcuts:

- `Ctrl/Cmd+P`: toggle profile
- `Ctrl/Cmd+L`: toggle track list
- `N`: new track
- `E`: enter or exit edit mode for the active track
- `S`: open selection mode
- `R`: rectangle selection mode
- `Delete` or `Backspace`: delete selected object or selected points
- `Ctrl/Cmd+D`: duplicate selection
- `Ctrl/Cmd+C`: copy selection
- `Ctrl/Cmd+X`: cut selection
- `Ctrl/Cmd+V`: paste into the current file or after the current object
- `Ctrl/Cmd+I`: open `Info`
- `Ctrl/Cmd+Shift+S`: open `Simplify...`
- `Esc`: close menu, cancel mode, or clear rectangle selection

Rules:

- Shortcuts must be disabled while typing in text inputs.
- On macOS, `Cmd` equivalents must work everywhere `Ctrl` is documented.
- If `Ctrl/Cmd+L` proves too conflicting with browser focus behavior, keep it as an app shortcut only when the map has focus and expose `Shift+L` as fallback.

## 7. Advanced Pause detection and cleanup

### Detection rule

A pause is a contiguous span where all included points remain within:

- 20 m horizontal radius
- 50 m vertical range
- 4 minutes minimum duration

This is stricter and more useful than the current simple by-default timestamp-gap rule.

### Detection output

For each pause, compute:

- start index
- end index
- duration
- barycenter coordinates
- mean elevation
- horizontal radius
- vertical range

### Cleanup action

Menu item: `Clean pauses...`

Base cleanup:

- Replace the full pause cluster with one representative point at the barycenter
- Preserve the first timestamp on entry and the last timestamp on exit on the adjacent non-pause points

Options:

- `Split segment before and after each cleaned pause`
- `Add waypoint at each cleaned pause`
- `Only add waypoints for pauses >= X min`

Default behavior:

- cleanup on
- split on
- waypoint off

## 8. Densify track

Menu item: `Add points every 5 m max`

Rules:

- Applies to tracks and segments
- Insert interpolated points so no adjacent points exceed 5 m horizontal spacing
- Interpolate elevation linearly when both neighbors have elevation
- Interpolate timestamps linearly when both neighbors have timestamps
- Do not change endpoints

This operation belongs in the same shared geometry-ops module as simplify and split.

## 8.1 Rectangle selection

The current rectangle delete feature should be generalized into `Rectangle selection`.

Flow:

- User enters rectangle selection mode.
- User drags a rectangle on the map.
- Points inside the rectangle are selected across the active editable track.
- The existing track action bar is reused instead of spawning a second action surface.
- While a rectangle selection exists, eligible actions in that bar switch into `selection context` styling, using a bright blue active state.
- A small anchored popup or hint chip near the rectangle explains what is selected and what the highlighted action will operate on.

Rectangle action UI:

- `Delete`
- `Split before`
- `Split after`
- `Simplify...`
- `Add intermediate points`
- `Cancel selection`

Rules:

- only one active editable track at a time.
- The selection model should store point indices, not duplicated coordinates.
- Rectangle selection should always resolve to the same `selectionSpan` structure used by the unified tool contract.
- If selected points are discontinuous, split actions operate on the smallest enclosing continuous index range and warn the user.
- Reusing the main action bar
- The anchored popup should stay informational, not become a second toolbar. It should show selection size, selected span, and a short consequence label such as `split around selected span` or `delete 42 points`.
- The selection context remains non-modal so the user can pan, zoom, inspect, and then apply an action.
- Unrelated actions in the bar should dim or disable while selection context is active.
- `Esc` clears the rectangle selection and returns the action bar to its normal state.

Profile linkage:

- As soon as a rectangle selection exists, the profile switches to the smallest continuous enclosing section of the active track that contains all selected points.
- If the rectangle intersects multiple disjoint spans, the profile still shows the smallest enclosing continuous span.
- The profile header must clearly indicate that it is showing a filtered section, with a one-click way to return to the full track.

## 9. Export compatibility profile

The exporter should move from minimal GPX output to a compatibility-oriented profile that stays inside GPX 1.1 core fields.

### Always emit at file level

- `metadata/name`
- `metadata/desc`
- `metadata/time`
- `metadata/bounds`

### Always emit for tracks and routes

- `name`
- `cmt`
- `desc`
- `type`

### Emit for waypoints when available

- `name`
- `cmt`
- `desc`
- `sym`
- `type`

Rules:

- Use empty strings for `desc` and `cmt` when the user left them blank and the compatibility profile is enabled.
- Do not emit empty `link`, `src`, or vendor-specific `extensions` in V1.
- Keep segment structure on export.
- Folder objects are never serialized directly; they map to filesystem grouping only.

## 10. Profile refinements

### 10.1 Horizontal speed smoothing

Use a time-based centered moving average after pause segmentation.

Default:

- 15 s minimum window
- expand to at least 20 m traveled when the sampling rate is sparse

Reason: horizontal speed is usually noisy but not as unstable as vertical speed.

### 10.2 Vertical speed smoothing

Use a pause-aware robust pipeline:

1. Split the track at detected pauses.
2. Build consecutive 15 s minimum and 20 m traveled minimum bins inside each non-pause span.
3. Compute raw vertical speed per bin in m/h.
4. Apply a rolling median over 5 bins.

Reason: this keeps change points visible while strongly reducing GPS elevation spikes.

### 10.3 Smoothed terrain slope

Add a new profile curve: `Smoothed terrain slope`.

Definition:

- sample terrain slope along the track as today
- compute a centered distance-window average over at least 20 horizontal meters
- shrink the window near segment ends instead of crossing a pause or segment boundary

Default display state: off.
